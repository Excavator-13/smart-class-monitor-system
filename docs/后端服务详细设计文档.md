## 1. 文档目的

本文档面向智慧教室项目的 SpringBoot 后端开发，基于当前后端接口文档，对后端服务的职责边界、模块划分、数据模型、接口实现、外部服务协作、关键流程、异常处理、安全约束和交付检查进行详细设计。

后端当前需要重点实现以下能力：

1. 用户登录与鉴权：支持 `admin`、`teacher` 角色登录、Token 校验和当前用户信息获取。
2. 结构化业务数据管理：管理视频源、人员、人脸特征元数据、区域配置、行为规则、告警事件、统计数据和操作日志。
3. 前端 REST API：向 Vue3 前端提供统一响应格式、统一分页格式和 Swagger/OpenAPI 文档。
4. AI 服务协作：向 AI 提供告警入库、配置拉取、人脸特征同步接口，并在人员注册、配置变更、健康检查时调用 AI 服务。
5. 流媒体资源协作：不代理视频流，只保存和返回 Nginx 静态资源相对路径，并解析 Nginx `/stat` 判断推流状态。

## 2. 与现有前端、AI、Nginx 设计的对齐结论

### 2.1 服务边界

| 服务 | 默认端口 | 前端环境变量 | 主要职责 |
| --- | --- | --- | --- |
| Vue 前端 | 5173/8081 等 | `VITE_API_BASE`、`VITE_AI_BASE`、`VITE_NGINX_BASE` | 页面展示、表单交互、视频播放、区域绘制、告警处理 |
| SpringBoot 后端 | 8080 | `VITE_API_BASE` | 鉴权、结构化数据管理、接口联调、告警流转、统计、Swagger 文档 |
| Flask AI 服务 | 5000 | `VITE_AI_BASE` | 视频帧分析、人脸识别、异常检测、处理后视频流输出、特征提取 |
| Nginx RTMP/静态资源 | 9090/9092 | `VITE_NGINX_BASE` | RTMP 推拉流、自动录像、截图和录像文件静态访问 |
| MySQL | 3306 | 无 | 后端业务数据持久化 |

### 2.2 SpringBoot 后端职责

SpringBoot 后端负责：

- 用户登录、JWT Token、角色区分和接口鉴权。
- 学生基础信息、人脸注册记录、特征向量元数据管理。
- 视频源信息管理和推流状态探活。
- 区域配置和行为规则配置管理。
- 接收 AI 确认告警并持久化，支持告警状态流转。
- 提供首页统计、告警统计、系统健康状态。
- 提供 Swagger/OpenAPI 文档和 DTO/VO 示例。

SpringBoot 后端不负责：

- 不运行 AI 模型，不进行视频帧推理。
- 不代理实时视频流，前端直接访问 Flask MJPEG 或 Nginx 资源。
- 不物理存储截图和录像文件，只保存 Nginx 静态目录下的相对路径。
- 不在 JSON 响应中硬编码服务器 IP。

### 2.3 服务调用关系

```
摄像头/OBS
  -> RTMP 推流
  -> Nginx RTMP 9090
  -> Flask AI 拉流分析
  -> AI 确认异常后 POST /alerts/ai
  -> SpringBoot 写入 MySQL
  -> Vue 前端 GET /alerts 查询和处理

Vue 前端
  -> SpringBoot: /auth /streams /students /zones /rules /alerts /alert-stats /recordings /system/health
  -> Flask AI: /video_feed/{stream_id}
  -> Nginx 9092: /snapshots/... /records/... /recordings/...

SpringBoot
  -> Flask AI: /face/feature/extract /face/features/reload /config/reload /model/status
  -> Nginx 9092: /stat
  -> MySQL: 持久化业务数据

Flask AI
  -> SpringBoot: /streams /zones /rules /students/face-features /alerts/ai
```

### 2.4 前端联调原则

前端通过环境变量访问不同服务：

```
VITE_API_BASE=http://localhost:8080
VITE_AI_BASE=http://localhost:5000
VITE_NGINX_BASE=http://39.106.209.208:9092
```

生产环境：

```
VITE_API_BASE=http://39.106.209.208:8080
VITE_AI_BASE=http://39.106.209.208:5000
VITE_NGINX_BASE=http://39.106.209.208:9092
```

后端必须遵守：

- JSON 字段使用 `snake_case`。
- 列表接口分页字段统一为 `page`、`page_size`、`records`、`total`。
- 截图、录像路径返回相对路径，例如 `/snapshots/20260707/phone_0001.jpg`。
- 不返回 [`http://39.106.209.208:9092/...`](http://39.106.209.208:9092/...) 这类硬编码绝对资源地址。
- SpringBoot 不代理 `/video_feed/{stream_id}`，只在 `/streams/{stream_id}/preview-url` 返回播放地址。

### 2.5 三端审查后的修订原则

本次根据“三端文档审查问题清单”补充以下设计约束：

- 告警入库接口统一按后端文档使用 `POST /alerts/ai`，不使用 `POST /alerts`。
- SpringBoot 调 AI 模型状态接口统一使用 `/model/status`，不使用 `/ai/model/status`。
- 人脸注册流程统一为“前端 → SpringBoot → Flask AI → SpringBoot 入库”，前端不直接调用 Flask 进行人脸注册。
- 人脸特征维度确认为 **512 维**（InsightFace ArcFace 模型）。数据库仍保存 `feature_dim` 字段以保持灵活性，但当前实现以 512 维为准。
- 前端上传人脸图片统一使用 base64；multipart 不作为初版方案。
- 候选事件和正式告警分开建模。AI 候选事件可有 `event_status=candidate`，SpringBoot 正式告警只使用 `unhandled`、`processing`、`handled`、`false_alarm`、`ignored`。
- AI 候选事件到 SpringBoot 告警入库需要明确字段映射，避免 `event_type`/`alert_type`、`target`/`target_info` 混淆。
- 实时推送如需实现，应走”AI → SpringBoot 入库 → SpringBoot WebSocket/SSE 推前端”，不采用 Flask 直接推前端正式告警。
- **已确认新增**：`fall_detected`（摔倒检测）、`danger_zone_approach`（危险区域接近预警）纳入本期告警类型枚举，同步更新接口、规则和数据库设计。
- **已确认新增**：区域配置增加 `safe_distance` 参数，用于接近预警距离阈值。
- **已确认新增**：`/rules/{id}` PUT 增加 `confidence_threshold` 和 `cooldown_seconds` 参数。
- **已确认新增**：告警入库请求体增加 `event_time_offset`（可选）顶级字段。
- **已确认变更**：告警详情返回 `record_url`（指向 Nginx 自动录像），不再使用 `video_clip_url`。
- 防欺骗认证、逐级上报、监控日报、轨迹追踪、异常声学事件等可选加分项，本期不实现，标注为后续扩展。

### 3.5 状态码设计

| code | 含义 | 典型场景 |
| --- | --- | --- |
| 0 | 成功 | 正常返回 |
| 400 | 参数错误 | 必填字段缺失、格式不正确、非法枚举 |
| 401 | 未登录或 Token 失效 | 前端需跳转登录 |
| 403 | 无权限 | 角色不允许访问 |
| 404 | 资源不存在 | ID 查询不到记录 |
| 409 | 状态冲突或重复数据 | 重复 `stream_id`、`student_no`、`event_id` |
| 500 | 服务端异常 | 未预期异常、外部服务不可用且无法降级 |

## 4. 后端内部模块划分

### 4.1 公共基础模块

职责：

- 统一 `ApiResponse<T>` 和 `PageResult<T>`。
- 统一异常处理，屏蔽堆栈细节。
- 统一枚举定义，包括角色、视频源状态、区域类型、告警类型、告警等级、告警状态。
- 配置 Jackson 使用 `SNAKE_CASE`、时区和日期格式。
- 配置跨域，允许前端开发端口请求。
- 配置 Swagger 分组和 DTO/VO 示例。

关键设计：

- Controller 不直接构造 `Map` 返回，统一返回 VO。
- `BusinessException` 携带业务状态码和错误信息。
- 参数校验错误统一转换为 `code=400`。
- 数据库唯一约束冲突统一转换为 `code=409`。

### 4.2 鉴权模块

涉及接口：

| 接口 | 方法 | 状态 | 说明 |
| --- | --- | --- | --- |
| `/auth/login` | POST | 已确定 | 用户登录，返回 token 和用户信息 |
| `/auth/info` | GET | 已确定 | 获取当前登录用户信息 |
| `/auth/logout` | POST | 待定 | 初版可空实现，前端也可直接清 token |

核心流程：

```
POST /auth/login
  -> 校验 username/password
  -> 查询 user
  -> 校验 password_hash
  -> 校验用户状态
  -> 生成 JWT
  -> 返回 token、用户信息、角色
```

设计要点：

- 密码不明文存储，数据库字段使用 `password_hash`。
- JWT 中至少包含 `user_id`、`username`、`role`、过期时间。
- 初版权限粒度为角色级，`admin` 可访问管理接口，`teacher` 可访问查询和告警处理接口。
- `/auth/login`、`/system/health` 可按需要放行，其余前端接口默认需要 Token。
- AI 专用接口使用单独的内部 Token 或签名，不与前端 JWT 混用。

### 4.3 视频源管理模块

涉及接口：

| 接口 | 方法 | 状态 | 说明 |
| --- | --- | --- | --- |
| `/streams` | GET | 已确定 | 视频源列表 |
| `/streams` | POST | 已确定 | 新增视频源 |
| `/streams/{stream_id}/status` | GET | 已确定 | 查询推流状态 |
| `/streams/enabled` | GET | 待定 | 获取已启用视频源 |
| `/streams/{id}` | GET | 待定 | 视频源详情 |
| `/streams/{id}` | PUT | 待定 | 编辑视频源 |
| `/streams/{id}` | DELETE | 待定 | 删除视频源 |
| `/streams/{stream_id}/preview-url` | GET | 待定 | 返回播放地址 |

核心数据：

- `id`：数据库主键，主要用于管理页详情、编辑、删除。
- `stream_id`：业务唯一标识，必须与推流端 Stream Key 一致。
- `stream_name`：展示名称。
- `rtmp_url`：格式为 `rtmp://39.106.209.208:9090/live/{stream_id}`。
- `status`：`enabled`、`disabled`、`online`、`offline`。

新增视频源流程：

```
POST /streams
  -> 校验 stream_id 唯一
  -> 校验 rtmp_url 格式
  -> 保存 video_stream
  -> 返回新增结果
```

推流状态检测流程：

```
GET /streams/{stream_id}/status
  -> 查询 video_stream
  -> NginxClient 请求 http://39.106.209.208:9092/stat
  -> 解析 XML 中 stream/name
  -> 查找 publish active="true"
  -> 返回 online/offline、推流时长、断流时间等
```

设计要点：

- 解析 `/stat` 失败时，不应影响视频源基础列表查询。
- `/streams/{stream_id}/status` 可返回 `offline` 并附带错误信息，不直接抛 500，除非后端自身异常。
- `preview-url` 只返回地址，不代理视频流。
- 删除视频源建议软删除，避免历史告警无法关联。
- `/streams` GET 返回数据中必须同时包含 `id` 和 `stream_id`。前端首页、AI 同步、推流状态查询使用 `stream_id`；管理页详情、编辑、删除使用 `id`。如果后续要统一为全量 `stream_id` 路径，需要同步前端文档和后端接口文档。

### 4.4 人员与人脸管理模块

涉及接口：

| 接口 | 方法 | 状态 | 说明 |
| --- | --- | --- | --- |
| `/students` | GET | 已确定 | 人员分页列表 |
| `/students` | POST | 已确定 | 新增人员 |
| `/students/{id}/face` | POST | 已确定 | 人脸注册 |
| `/students/{id}` | GET | 待定 | 人员详情 |
| `/students/{id}` | PUT | 待定 | 编辑人员 |
| `/students/{id}` | DELETE | 待定 | 删除人员 |
| `/students/{id}/face-features` | GET | 待定 | 查看特征元数据 |
| `/students/{id}/face-features/{feature_id}` | DELETE | 待定 | 删除特征 |
| `/students/face-features` | GET | 待定 | AI 同步人脸特征库，禁止前端访问 |

新增人员流程：

```
POST /students
  -> 校验 student_no 唯一
  -> 校验 name/class_name
  -> 插入 student
  -> face_registered 默认 false
```

人脸注册流程：

```
POST /students/{id}/face
  -> 校验 student 存在且启用
  -> 校验 base64 图片格式和大小
  -> AiClient 调用 POST {AI_BASE_URL}/face/feature/extract
  -> AI 返回 face_count、feature_dim、feature_vector、quality、bbox
  -> 校验 face_count=1
  -> 保存 face_feature，包括 feature_dim 和 feature_vector
  -> 更新 student.face_registered = true
  -> 调用 POST {AI_BASE_URL}/face/features/reload
  -> 返回注册结果和质量评分
```

设计要点：

- 前端接口为 `/students/{id}/face`，不使用 `/students/{id}/face-register`。
- 初版人脸图片传输统一使用 base64，与 SpringBoot 调 AI 的 `/face/feature/extract` 保持一致；multipart 暂不作为默认方案。
- 特征向量只对 AI 内部接口返回，不对前端开放。
- `/students/{id}/face-features` 只返回元数据，例如图片路径、创建时间、版本、质量评分，不返回完整 `feature_vector`。
- 特征向量初版可用 MySQL JSON 或 MEDIUMTEXT 保存，后续可迁移向量库。
- 后端不假设固定特征维度，以 AI 返回的 `feature_dim` 为准。当前 AI 技术选型确认为 InsightFace（512 维），数据库 `feature_dim` 字段保存实际维度值。
- 删除学生建议软删除，并同步处理或标记关联特征不可用。

### 4.5 区域配置模块

涉及接口：

| 接口 | 方法 | 状态 | 说明 |
| --- | --- | --- | --- |
| `/zones` | GET | 已确定 | 查询区域配置 |
| `/zones` | POST | 待定 | 新增区域 |
| `/zones/{id}` | GET | 待定 | 区域详情 |
| `/zones/{id}` | PUT | 待定 | 修改区域 |
| `/zones/{id}` | DELETE | 待定 | 删除区域 |
| `/streams/{stream_id}/zones` | GET | 待定 | 获取某视频源全部区域 |

区域坐标格式：

```json
{
  "coordinates": [
    {"x": 0.12, "y": 0.18},
    {"x": 0.48, "y": 0.18},
    {"x": 0.46, "y": 0.52},
    {"x": 0.10, "y": 0.50}
  ]
}
```

设计要点：

- 坐标必须使用 0 到 1 的归一化比例，不存像素坐标。
- `zone_type` 使用既有枚举：`danger`、`seat`、`phone_forbidden`、`roi`。
- 修改区域后应通过 `AiClient` 调用 `/config/reload` 通知 AI 刷新。
- 删除区域建议软删除，避免历史告警中的 `zone_id` 无法追溯。
- **已确认**：区域配置增加 `safe_distance` 参数（人员距边缘低于此值时触发 `danger_zone_approach` 接近预警）。`/zones` POST/PUT 入参包含 `safe_distance`，区域表增加对应字段。`danger_zone_approach` 已纳入告警类型枚举。

### 4.6 行为规则模块

涉及接口：

| 接口 | 方法 | 状态 | 说明 |
| --- | --- | --- | --- |
| `/rules` | GET | 已确定 | 查询规则列表 |
| `/rules/{id}` | PUT | 已确定 | 更新规则 |
| `/rules/{id}` | GET | 待定 | 规则详情 |
| `/rules` | POST | 待定 | 新增规则 |
| `/rules/{id}` | DELETE | 待定 | 删除规则 |

规则字段：

- `rule_type`：例如 `phone_usage`、`flame_detected`、`danger_zone_intrusion`。
- `enabled`：是否启用。
- `threshold_seconds`：持续时间阈值。
- `confidence_threshold`：置信度阈值。
- `cooldown_seconds`：冷却时间。
- `config_json`：扩展配置。

设计要点：

- 初版可预置全局规则，后续再扩展为按 `stream_id` 或场景模板。
- 阈值不在前端写死，前端只展示和提交配置。
- 更新规则后通知 AI `/config/reload`。
- 规则删除初版可不开放，避免 AI 侧配置缺失。
- **已确认**：`/rules/{id}` PUT/POST 入参包含 `confidence_threshold` 和 `cooldown_seconds`，作为独立字段（非仅从 `config_json` 读取）。AI 端规则结构和数据库表均有对应列。
- **已确认**：本期新增 `fall_detected`（摔倒检测）和 `danger_zone_approach`（接近预警）两种规则类型，初版需预置对应规则记录。

### 4.7 告警管理模块

涉及接口：

| 接口 | 方法 | 状态 | 说明 |
| --- | --- | --- | --- |
| `/alerts` | GET | 已确定 | 告警分页列表 |
| `/alerts/{id}` | GET | 已确定 | 告警详情 |
| `/alerts/{id}/status` | PUT | 已确定 | 告警状态流转 |
| `/alert-stats` | GET | 已确定 | 首页统计卡片 |
| `/alerts/ai` | POST | 待定 | AI 告警入库 |
| `/alerts/{id}/feedback` | POST | 待定 | 误报反馈 |
| `/alerts/export` | GET | 扩展 | 导出告警 |

AI 告警入库流程：

```
POST /alerts/ai
  -> 校验内部访问凭证
  -> 校验 event_id、stream_id、alert_type、level、occurred_at
  -> 根据 event_id 查询 alert_event
  -> 已存在则直接返回已有 alert_id 和 status
  -> 不存在则插入 alert_event，status=unhandled
  -> 返回 alert_id、status
```

告警状态流转：

```
unhandled
  -> processing
  -> handled

unhandled
  -> false_alarm

unhandled
  -> ignored

processing
  -> handled / false_alarm / ignored
```

设计要点：

- `event_id` 必须加唯一索引，保证 AI 重试不重复入库。
- `snapshot_path`、`record_path` 保存相对路径。
- AI 入库字段使用 `snapshot_path`、`record_path`；SpringBoot 对前端返回时映射为 `snapshot_url`、`record_url`，二者都只返回相对路径。
- 初版没有独立视频切片能力时，`record_url` 指向 Nginx 自动录像文件的相对路径（如 `/classroom_01-2026-07-07-10_30_00.mp4`）。不要虚构 `/clips/` 地址，除非 AI/Nginx 已实际生成切片。
- 状态更新时记录处理人、处理时间、备注。
- 标记误报后可继续调用 `/alerts/{id}/feedback` 记录原因，但此接口当前为待定。
- `stream_offline` 职责分工：SpringBoot 通过 Nginx `/stat` 低频探活，用于展示视频源在线状态；AI 拉流失败生成 `stream_offline` 告警，用于实时告警入库。两者不一致时，告警判断以 AI 实时拉流状态为准，页面状态展示以 SpringBoot 探活为准并提示来源。
- 当前告警类型按后端接口文档 v2.1 枚举实现，已确认包含 `face_recognized`、`stranger_detected`、`danger_zone_intrusion`、`danger_zone_stay`、`danger_zone_approach`、`phone_usage`、`head_down`、`leave_seat`、`flame_detected`、`fall_detected`、`stream_offline`。`spoofing_attack`、`acoustic_anomaly` 等可选加分项属于后续扩展，本期不实现。

## 5. 数据模型详细设计

### 5.1 数据库设计原则

- 表名沿用后端接口文档：`user`、`student`、`face_feature`、`video_stream`、`danger_zone`、`behavior_rule`、`alert_event`、`operation_log`。
- 字段统一使用 `snake_case`，Java Entity 使用驼峰命名并由 MyBatis 映射。
- 所有配置类表建议保留 `created_at`、`updated_at`、`deleted_at`，支持软删除和历史追溯。
- 所有外部资源路径只保存相对路径，例如 `/snapshots/20260707/xxx.jpg`。
- JSON 类字段优先使用 MySQL `JSON` 类型；若部署版本不支持，可退化为 `TEXT` 并在 Service 层校验 JSON 格式。
- 大字段如 `feature_vector` 可使用 `JSON` 或 `MEDIUMTEXT`。后端必须保存 `feature_dim`，不假设固定 128 或 512 维。
- 金额无关，时间字段统一使用 `DATETIME`，业务时间按 `Asia/Shanghai` 处理。

### 6.1 接口命名和路径约定

当前阶段以后端接口文档和前端接口文档为联调基准，路径暂不强制添加 `/api/v1` 前缀。

```
/rules
/alerts
/alert-stats
/recordings

```

AI 调用 SpringBoot 的专用接口包括：

```

```

### 6.2 DTO/VO 设计原则

请求 DTO：

- 字段使用 Java 驼峰命名，由 Jackson 输出为 `snake_case`。
- 使用 `@NotBlank`、`@NotNull`、`@Min`、`@Max`、`@Pattern` 等注解做基础校验。
- Swagger `@Schema` 必须写清字段含义、是否必填、示例值。

响应 VO：

- 不直接返回 Entity。
- 不返回敏感字段，例如 `password_hash`、完整人脸向量给前端。
- 文件字段只返回相对路径。
- 分页列表统一封装为 `PageResult<T>`。

### 6.3 重点接口行为

#### `/alerts/ai`

请求字段以接口文档为准：

```json
{
  "event_id": "evt_20260707102030001",
  "stream_id": "classroom_01",
  "alert_type": "flame_detected",
  "alert_name": "明火检测",
  "level": "high",
  "confidence": 0.91,
  "occurred_at": "2026-07-07T10:20:30+08:00",
  "duration_seconds": 1.5,
  "student_id": null,
  "target_info": {
    "track_id": "fire_1",
    "bbox": [300, 120, 380, 240]
  },
  "zone_id": null,
  "snapshot_path": "/snapshots/20260707/evt_20260707102030001.jpg",
  "record_path": "/records/20260707/classroom_01.flv",
  "event_time_offset": 12.5,
  "extra": {
    "model": "yolo_fire_v1",
    "rule": "flame_confidence_duration"
  }
}
```

实现要求：

- 校验 `event_id`、`stream_id`、`alert_type`、`level`、`occurred_at`。
- `snapshot_path`、`record_path` 如果存在，必须以 `/` 开头，不允许包含服务器 IP。
- 相同 `event_id` 重试时直接返回已有告警。
- 默认状态为 `unhandled`。
- `event_time_offset` 已确认作为可选顶级字段，表示事件在录像文件中的秒级偏移。
- `student_id` 当前接口文档示例类型为 string，数据库建议保存为 [`student.id`](http://student.id)。联调前需确认 AI 传的是学生业务编号 `student_no` 还是数据库主键；若传 `student_no`，SpringBoot 入库前负责转换。

返回：

```json
{
  "code": 0,
  "message": "success",
  "data": {
    "alert_id": 10086,
    "status": "unhandled"
  }
}
```

AI 候选事件到正式告警入库字段映射：

| AI 候选事件字段 | `/alerts/ai` 字段 | 入库字段 | 说明 |
| --- | --- | --- | --- |
| `event_id` | `event_id` | `alert_event.event_id` | 保持不变，用于幂等 |
| `event_type` | `alert_type` | `alert_event.alert_type` | 命名转换 |
| `event_name` | `alert_name` | `alert_event.alert_name` | 命名转换 |
| `status=candidate/confirmed` | 不传 | `alert_event.status=unhandled` | 候选事件状态不进入正式告警状态 |
| `stream_id` | `stream_id` | `alert_event.stream_id` | 保持不变 |
| `level` | `level` | `alert_event.level` | 保持不变 |
| `confidence` | `confidence` | `alert_event.confidence` | 保持不变 |
| `occurred_at` | `occurred_at` | `alert_event.occurred_at` | 保持不变 |
| `duration_seconds` | `duration_seconds` | `alert_event.duration_seconds` | 保持不变 |
| `target.student_id` | `student_id` | `alert_event.student_id` | 建议传数据库主键；若传业务编号需转换 |
| `target.track_id`、`target.bbox` | `target_info` | `alert_event.target_info` | 目标信息合并为 JSON |
| `zone.zone_id` | `zone_id` | `alert_event.zone_id` | 只入库区域 ID，名称可通过关联查询 |
| `snapshot_path` | `snapshot_path` | `alert_event.snapshot_path` | 相对路径 |
| `record_path` 或 `record_file_path` | `record_path` | `alert_event.record_path` | 统一为 `record_path` |
| `event_time_offset` | `event_time_offset` | `alert_event.event_time_offset` | 可选顶级字段，已确认 |
| `extra` | `extra` | `alert_event.extra` | 保存模型、规则等扩展信息 |

#### `/students/{id}/face`

实现要求：

- 初版统一使用 base64 图片，与 SpringBoot 调 AI `/face/feature/extract` 的请求体保持一致。
- 请求体建议包含 `image`，可选 `image_type=base64`；是否支持 multipart 作为后续扩展确认。
- 后端校验学生存在后调用 AI `/face/feature/extract`。
- AI 返回未检测到人脸、多个人脸、图片无法解析等错误时，转换为统一响应。
- 成功后保存特征并更新 `face_registered`。
- 保存后通知 AI `/face/features/reload`，通知失败可记录日志并返回注册成功，具体是否阻断需联调确认。

#### `/streams/{stream_id}/status`

实现要求：

- 不读取数据库状态直接判断在线，而是请求 Nginx `/stat`。
- 返回在线状态、推流时长、断流时间等。
- Nginx 状态页不可访问时，返回可解释的离线或未知状态。

#### `/alerts/{id}/status`

实现要求：

- 校验告警存在。
- 校验目标状态在枚举范围内。
- 记录处理人、处理时间、备注。
- 不允许无意义重复更新时，可返回成功或 `409`，具体按前端体验确定。

## 7. 核心业务流程详细设计

### 7.1 登录流程

```
前端提交 username/password
  -> AuthController.login
  -> AuthService 校验用户存在
  -> 校验密码 hash
  -> 校验用户状态
  -> JwtTokenProvider 生成 token
  -> 返回 token、user_id、username、role、nickname/avatar
```

异常处理：

- 用户不存在或密码错误，返回 `code=400` 或统一登录失败信息。
- 用户停用，返回 `code=403`。
- 参数缺失，返回 `code=400`。

### 7.2 实时视频分析到告警入库流程

```
摄像头/OBS 推流到 Nginx
  -> Flask AI 拉取 RTMP 流
  -> AI 从 SpringBoot 拉取 streams/zones/rules/face-features
  -> AI 检测到异常并完成去抖
  -> AI 保存截图到 Nginx 静态目录
  -> AI 调 POST /alerts/ai
  -> SpringBoot 按 event_id 幂等入库
  -> 前端轮询 GET /alerts 展示
  -> 教师或管理员处理告警
```

后端关键点：

- 入库只接收确认告警，不接收每帧数据。
- `event_id` 唯一索引用于 AI 重试幂等。
- 路径字段不拼接 `VITE_NGINX_BASE`，只保存相对路径。

### 7.3 人脸注册流程

```
管理员先 POST /students 创建人员
  -> 前端上传人脸图片到 POST /students/{id}/face
  -> SpringBoot 校验人员和图片
  -> SpringBoot 调 AI /face/feature/extract
  -> AI 返回 feature_dim、feature_vector、质量评分、bbox
  -> SpringBoot 写入 face_feature
  -> SpringBoot 更新 student.face_registered = true
  -> SpringBoot 调 AI /face/features/reload
  -> 前端展示注册结果
```

后端关键点：

- 前端不直接调用 AI 提取特征。
- SpringBoot 是人脸特征入库责任方。
- AI 通过 `/students/face-features` 获取全量或增量特征库。

### 7.4 配置变更刷新 AI 流程

```
管理员修改 streams/zones/rules
  -> SpringBoot 保存数据库
  -> SpringBoot 调 AI /config/reload
  -> AI 重新拉取配置
  -> SpringBoot 返回前端更新结果
```

设计要点：

- 保存数据库和通知 AI 是两个动作。
- 如果通知 AI 失败，应记录日志，可在响应中提示配置已保存但 AI 刷新失败。
- 推模式优先：SpringBoot 在视频源、区域、规则保存成功后主动调用 AI `/config/reload`。
- 拉模式兜底：AI 定时调用 `/streams`、`/zones`、`/rules` 拉取配置，防止通知失败后配置长期不一致。
- 人脸特征变更使用独立刷新接口 `/face/features/reload`，不要和区域/规则配置刷新混在一个语义里。
- `/config/reload` 失败不应回滚已经提交的数据库事务，但需要记录操作日志或系统日志，便于手动重试。

### 7.5 告警状态处理流程

```
AI 写入告警，状态 unhandled
  -> 前端查看列表和详情
  -> 教师/管理员 PUT /alerts/{id}/status
  -> SpringBoot 更新状态、处理人、处理时间、备注
  -> 如为 false_alarm，可继续 POST /alerts/{id}/feedback
```

状态枚举：

```
unhandled
processing
handled
false_alarm
ignored
```

### 7.6 推流状态检测流程

```
前端请求 GET /streams/{stream_id}/status
  -> SpringBoot 查询视频源存在性
  -> 请求 Nginx 9092 /stat
  -> 解析 XML
  -> 返回 online/offline/unknown
```

设计要点：

- 解析逻辑放在 `NginxClient` 或独立解析器中。
- 不在 Controller 中解析 XML。
- 状态页异常时要有明确降级返回。

## 8. 异常处理与降级策略

| 场景 | 处理策略 |
| --- | --- |
| 参数校验失败 | 返回 `code=400`，提示具体字段 |
| Token 缺失或过期 | 返回 `code=401` |
| 角色权限不足 | 返回 `code=403` |
| 资源不存在 | 返回 `code=404` |
| 唯一字段重复 | 返回 `code=409` |
| AI 特征提取失败 | 保留 AI 错误语义，转换为统一响应 |
| AI 刷新缓存失败 | 记录日志，可按场景返回成功加提示或业务错误 |
| Nginx `/stat` 不可用 | 推流状态返回 `unknown` 或 `offline`，健康检查标记 RTMP 异常 |
| 数据库异常 | 返回 `code=500`，日志记录 SQL 和上下文 |
| AI 重复推送同一告警 | 根据 `event_id` 返回已有 `alert_id`，不重复插入 |
| 截图或录像路径格式错误 | 返回 `code=400`，拒绝保存绝对 URL |

## 9. 安全与权限设计

### 9.1 前端接口鉴权

- 除登录、健康检查等公开接口外，前端业务接口默认需要 JWT。
- Token 通过 Header 传递，具体 Header 名称按前端约定实现。
- 初版角色控制：

    - `admin`：配置、人员、视频源、规则、告警处理、日志查询。
    - `teacher`：告警查看和处理、统计查看、基础查询。

### 9.2 AI 专用接口安全

AI 专用接口包括：

- `POST /alerts/ai`
- `GET /students/face-features`

安全要求：

- 生产环境限制内网访问。
- 使用内部 Token 或签名校验。
- 不允许前端访问 `/students/face-features`。
- 特征向量不出现在前端接口响应中。

### 9.3 文件路径安全

- 只接受以 `/` 开头的相对路径。
- 不接受 `http://`、`https://`、`file://` 等绝对路径。
- 不接受包含 `..` 的路径。
- 后端不拼接服务器 IP 返回给前端。

### 9.4 操作审计

建议记录以下操作：

- 登录成功和失败。
- 新增、编辑、删除视频源。
- 新增、编辑、删除人员。
- 人脸注册和特征删除。
- 新增、编辑、删除区域。
- 修改规则。
- 告警状态处理。
- 用户管理操作。

## 11. 开发优先级建议

### 第一阶段：MVP 核心链路

- SpringBoot + MyBatis + MySQL 基础工程。
- Swagger / springdoc-openapi。
- 统一响应、分页、异常处理。
- JWT 简化版鉴权。
- `/auth/login`、`/auth/info`。
- `/streams` GET/POST。
- `/streams/{stream_id}/status`。
- `/students` GET/POST。
- `/zones` GET。
- `/rules` GET、`/rules/{id}` PUT。
- `/alerts` GET、`/alerts/{id}` GET、`/alerts/{id}/status` PUT。
- `/alert-stats`。
- `/system/health`。

### 第二阶段：AI 联调和完整 CRUD

- `/students/{id}/face`，对接 AI `/face/feature/extract`。
- `POST /alerts/ai`，支持幂等入库。
- `GET /students/face-features`，支持 AI 同步特征向量。
- 视频源、人员、区域、规则完整 CRUD。
- `/recordings`。
- 配置变更后通知 AI `/config/reload`。
- 人脸特征变更后通知 AI `/face/features/reload`。
- `/rules/{id}` PUT 支持 `confidence_threshold` 和 `cooldown_seconds`。
- 告警详情中明确 `snapshot_path` 到 `snapshot_url`、`record_path` 到 `record_url` 的映射。
- 区域配置支持 `safe_distance` 参数（用于 `danger_zone_approach` 接近预警）。
- 预置 `fall_detected`、`danger_zone_approach` 规则记录。

### 第三阶段：完善和扩展

- `/operation-logs` 和操作日志 AOP。
- `/users` 用户管理 CRUD。
- WebSocket / SSE 实时告警推送。
- 告警导出 Excel/CSV。
- 更细粒度权限控制。
- 告警反馈和模型优化闭环。
- 活体/防欺骗、逐级上报、监控日报、行动轨迹追踪、异常声学事件等可选加分项按团队取舍进入扩展阶段。

## 14. 发现的不一致点

以下内容来自三端文档审查，本设计已作出处理或标注：

| # | 不一致点 | 本设计处理 |
| --- | --- | --- |
| 1 | AI 文档告警入库写 `POST /alerts`，后端文档写 `POST /alerts/ai` | 采用 `POST /alerts/ai` |
| 2 | AI 文档出现 `/ai/model/status`，后端文档为 `/model/status` | 采用 `/model/status` |
| 3 | AI 文档出现 `/ai/face/feature/extract`，后端文档为 `/face/feature/extract` | 采用 `/face/feature/extract` |
| 4 | AI 文档描述前端直调 Flask 注册人脸，后端文档描述 SpringBoot 主导 | 采用“前端 → SpringBoot → AI → SpringBoot 入库” |
| 5 | 人脸特征维度存在 128/512 冲突 | 已确认 InsightFace 512 维；后端保存 `feature_dim`，不硬编码 |
| 6 | AI 候选事件 `status=candidate` 与 SpringBoot 告警状态冲突 | 候选事件状态不入库，正式告警从 `unhandled` 开始 |
| 7 | `snapshot_path` 与 `snapshot_url` 命名不同 | 入库字段为 `snapshot_path`，前端返回字段可映射为 `snapshot_url` |
| 8 | AI 文档 `record_file_path` 与后端 `record_path` 不一致 | 统一为 `record_path` |
| 9 | `/streams/{stream_id}/status` 与 `/streams/{id}` 混用 | 明确 `stream_id` 用于业务状态，`id` 用于管理 CRUD，列表返回二者 |
| 10 | `/rules/{id}` PUT 缺少 `confidence_threshold`、`cooldown_seconds` | 已确认纳入 PUT/POST 入参 |
| 11 | `video_clip_url` 示例指向 `/clips/`，但初版无切片能力 | 已确认改用 `record_url` 指向 Nginx 自动录像 |
| 12 | 前端文档建议 Flask 直接 WebSocket 推告警 | 本设计要求实时推送由 SpringBoot 在入库后推送 |
| 13 | `stream_offline` 由 AI 和 SpringBoot 双重检测 | 明确 SpringBoot 用于展示探活，AI 用于实时告警 |
| 14 | 任务清单包含危险区域接近预警、摔倒、防欺骗等 | 危险区域接近预警、摔倒检测已确认纳入本期；防欺骗等标注为扩展 |
| 15 | `AGENTS.md` 提到 `/internal/ai/**` 和 `event_uid` | 当前接口文档仍使用 `/alerts/ai` 和 `event_id`，本文档不擅自改路径或字段名 |
