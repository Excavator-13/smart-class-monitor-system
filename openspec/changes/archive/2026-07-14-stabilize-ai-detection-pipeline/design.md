## Context

AI 服务目前由每个 `/video_feed/<stream_id>` 生成器循环读取 `StreamManager` 中的最近一帧并直接执行分析。缓存帧没有序号，生成器的循环速度也不受新帧到达约束，因此同一帧会被重复分析；推流停止而 `cap.read()` 尚未返回失败时，旧帧仍可推进事件持续时间并生成截图。失败后若 RTMP/OpenCV 短暂吐出缓冲帧，当前实现又会立即清空 `failure_started_at` 和 `offline_alerted`，把同一次断流拆成两次。

区域方面，YOLO 的 `bbox` 是像素坐标，前端保存的区域是 0–1 归一化坐标。`ZoneService` 和 `BehaviorService` 当前直接比较两者，测试却只构造了归一化 bbox。危险区事件还被排除在 `VISIBLE_DETECTION_EVENT_TYPES` 之外。明火方面，数据库规则使用 `fire_detected`，AI 事件使用 `flame_detected`，规则查找失败后回退到模型阈值 0.25；模型输出类别也未校验。

## Goals / Non-Goals

**Goals:**

- 每个捕获到的新帧在一个 AI 进程内最多推进一次事件状态，旧帧不得在断流后继续生成异常或截图。
- 同一次持续断流只确认一次离线事件；短暂缓冲帧不能被当成恢复，稳定新帧到达后才开启新周期。
- 在真实像素检测框下正确命中归一化手机禁用区和危险区，并输出可见画框及告警提示。
- 让明火检测使用当前启用的后端规则阈值并过滤非火焰类别，显著降低误报。

**Non-Goals:**

- 不重新训练或替换 YOLO 权重，不承诺仅靠规则过滤消除所有视觉模型误报。
- 不调整前端区域创建接口、数据库表或告警入库 JSON 契约。
- 不把分析循环整体迁移为独立后台工作进程；本次保留现有 MJPEG 驱动架构。

## Decisions

### 1. 为捕获帧增加单调序号和全局分析认领

`StreamState` 在发布可消费帧时递增 `frame_sequence`。视频生成器读取 `(frame, sequence)`，仅当 `StreamManager.claim_frame_for_analysis(stream_id, sequence)` 成功时执行 `analyze_frame`；重复轮询和其他消费者只编码当前画面，不再推进事件状态。

选择全局认领而不是仅在每个生成器保存 `last_sequence`，因为后者仍会让多个浏览器对同一帧重复分析。保留 `get_frame()` 兼容既有调用和测试，新增带序号读取方法。

### 2. 失败周期采用“稳定恢复”门控

第一次读取失败立即停止发布可分析帧并开始失败周期。重新打开捕获后，成功读取的帧先进入恢复候选，只有连续成功达到 `recovery_after_seconds` 才清除失败周期并发布新帧；候选期间不分析、不截图，也不清除 `offline_alerted`。这能隔离推流关闭后短暂吐出的 RTMP 缓冲。

相比查询 Nginx stat，此方案不增加 AI→Nginx/后端依赖，也适用于摄像头 RTSP 等来源。代价是恢复画面会延迟数秒，配置项允许部署调整。

### 3. 区域计算统一使用归一化空间，绘制保留像素空间

`AnalysisService` 从 `frame.shape` 得到宽高，并把尺寸显式传入行为和区域服务。服务在几何判断时将 bbox 中心/脚点除以宽高；返回事件的 `target.bbox` 仍保持原始像素值供 OpenCV 画框和前端使用。若测试或调用方传入的 bbox 已在 0–1 范围，则视为已归一化，保持兼容。

相比把区域多边形放大成像素，统一到归一化空间能让 `safe_distance` 继续保持现有 0–1 语义。

### 4. 区域事件同时进入目标画框与告警覆盖层

将 `danger_zone_intrusion`、`danger_zone_stay`、`danger_zone_approach` 加入可见检测类型，并提供中文/英文稳定标签。检测一出现即可画目标框；达到事件阈值并确认后继续使用现有左上角短时告警覆盖层。

### 5. 明火规则别名、有效阈值与类别白名单

明火服务优先读取 `flame_detected`，兼容读取后端现有 `fire_detected`。最终置信度取模型配置阈值和启用规则阈值的较大值，避免后端规则缺失或过低导致放宽模型安全下限。模型类别名必须命中配置的 `allowed_classes`（默认 `fire`, `flame` 及中文等价名）；模型没有可用类别元数据时保留单类专用权重的兼容路径。

明火仍通过 `threshold_seconds` 连续观察确认；track key 使用量化后的框位置而不是每帧数组序号，便于后续追踪，但事件服务当前按流和事件类型聚合，因此不改变接口语义。

## Risks / Trade-offs

- [稳定恢复窗口会延迟真正复流后的画面] → 默认采用较短且可配置的窗口，并通过连续成功帧测试验证；现场可按 RTMP 缓冲长度调节。
- [仅在有 MJPEG 消费者时执行分析的既有架构仍然存在] → 本次不扩展架构范围，但帧认领保证消费者数量不会改变告警频率。
- [火焰类别命名因自训练权重不同而变化] → 提供可配置白名单，并在无类别元数据的单类模型上兼容；状态日志暴露过滤配置。
- [历史测试使用归一化 bbox] → 坐标辅助函数自动识别 0–1 bbox，并新增像素 bbox 回归用例。

## Migration Plan

1. 部署 AI 代码和新增配置默认值，重启 Flask AI 进程以重建流状态。
2. 保持 SpringBoot、数据库与前端版本不变；区域配置通过现有 `/config/reload` 生效。
3. 用真实推流分别验证手机禁用区、危险区、无火场景和断流/复流。
4. 若现场复流延迟不可接受，可仅回调 `recovery_after_seconds`；代码回滚不涉及数据迁移。

## Open Questions

- 现场 `yolo_fire.pt` 的类别名称需要在部署验证时记录；若不是默认白名单，应在 `model.yaml` 明确配置，而不是放宽为接受所有类别。
